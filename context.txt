## 🧠 Contexto do Projeto: Baixada Vacinada

**Baixada Vacinada** é uma plataforma web (frontend + backend) para gestão e transparência das ações de vacinação na Baixada Fluminense. O projeto tem viés **social e comunitário**, e envolve funcionalidades para usuários públicos e administradores.

* **Usuários públicos** acessam informações sobre unidades de saúde, vacinas disponíveis, registros de vacinação e recebem notificações.
* **Administradores** podem gerenciar unidades de saúde (UBS), vacinas, registros e enviar notificações (ex: via WhatsApp).

A stack envolve:

* **Backend**: Node.js + Express + TypeScript
* **Banco de dados**: MongoDB Atlas
* **Deploy**: Vercel
* **API externa anteriormente usada**: [API do DataSUS](https://apidadosus.saude.gov.br/ubs), mas foi substituída por persistência em banco de dados local por problemas de confiabilidade e estrutura
* **Documentação**: Swagger via OpenAPI (`swagger.json`)
* **Organização de código** por domínios: `controllers/`, `routes/`, `usecases/`, `infrastructure/database/models/`, `domain/entities/` etc.

## ✅ O que já foi implementado

### 1. **Infraestrutura do projeto**

* Projeto backend criado com TypeScript.
* Estrutura de diretórios modularizada (separação entre domínio, controllers, routes, infraestrutura).
* Deploy Vercel testado com sucesso.
* MongoDB Atlas conectado e populado com collections:

  * `health_units`
  * `vaccines`
  * `users`

### 2. **Modelos e coleções**

* Modelo `HealthUnit` (antigo `ubsModel`) criado com os campos:

  ```ts
  {
    _id,
    name,
    address,
    city,
    state,
    zipCode
  }
  ```

* Em processo de **expansão** para contemplar:

  * `active: boolean`
  * `isFavorite: boolean`
  * `latitude` e `longitude` (para geolocalização)
  * `neighborhood` (bairro)
  * `schedule` e `phone` (com base em prints anteriores)

### 3. **Endpoints públicos**

* Endpoint GET `/public/healthunits` criado e atualizado para:

  * Buscar UBS diretamente do banco de dados (não mais da API do DataSUS).
  * Aceitar filtros via query params: `active`, `isFavorite`, `lat`, `lng`, `neighborhood`.

### 4. **Use Case: Listagem com filtros**

* `ListHealthUnitsUseCase` implementado com:

  * Interface `ListHealthUnitsFilters`
  * Método `buildQuery` para montar dinamicamente filtros do MongoDB a partir dos query params

### 5. **Controller público**

* Refatorado para utilizar o novo use case.
* Retorna corretamente erros 500 em caso de falhas com logs padronizados.

### 6. **Controller de administração**

* Implementado endpoint `POST /admin/healthunits` com persistência no MongoDB.
* Em processo de expansão para incluir:

  * `GET` por ID
  * `PUT` para atualização geral
  * `PATCH` para atualizar campos específicos como `active` e `isFavorite`

### 7. **Swagger**

* Estrutura do Swagger iniciada.
* Endpoints públicos e de admin agora padronizados para `/healthunits` (antes havia conflito entre `ubs` e `healthunits`).
* Documentação atualizada com base nos novos métodos e modelos.
